# Архитектура системы

## Компоненты

| Компонент | Роль |
|---|---|
| 3 источника заказов | Посетитель в зале, сайт кафе, телефонный звонок |
| **Агент 1** | Приём заказов, ведение статусов, ротация листов/таблиц по дням и месяцам |
| Таблица заказов (`orders_<месяц>`) | "Исходная" таблица, лист = текущий день |
| **Агент 2** | Каждые 30 минут пересчитывает заказы по блюдам и обновляет список для кухни |
| Таблица `kitchen assistant_<месяц>` | Рабочий список кухни, лист = текущий день |
| Кухня | Готовит по списку, вручную проставляет фактический статус |

## Общая схема

```mermaid
flowchart TB
    subgraph SRC["Источники заказов"]
        direction LR
        A1["🧍 Посетитель в зале"]
        A2["🌐 Сайт кафе"]
        A3["📞 Телефон"]
    end

    subgraph AG1["Агент 1 — приём и маршрутизация"]
        direction TB
        B1["Записать заказ\nстатус = new"]
        B2["Создать лист дня\n«16 July» и т.п."]
        B3["Создать новую таблицу\nв начале месяца"]
        B4["После подтверждения Агента 2:\nnew → redirected"]
    end

    OS[("📄 Таблица заказов\norders_July\nлист = текущий день")]

    subgraph AG2["Агент 2 — сводка для кухни (каждые 30 мин)"]
        direction TB
        C1["Прочитать строки\nсо статусом new"]
        C2["Просуммировать\nQuantity по блюду"]
        C3{"Проверить столбец C\nTotal status"}
        C4["C = done →\nB = новая сумма\nC = new"]
        C5["C = число →\nB = число + новая сумма"]
        C6["Новое блюдо →\nдобавить строку, C = new"]
        C7["Отсортировать:\nnew/число сверху, done снизу"]
    end

    KS[("📋 Kitchen assistant\nлист = текущий день\nA: блюдо · B: кол-во · C: статус")]

    K["👨‍🍳 Кухня"]

    A1 --> B1
    A2 --> B1
    A3 --> B1
    B1 --> OS
    B2 -.-> OS
    B3 -.-> OS

    OS --> C1 --> C2 --> C3
    C3 -->|done| C4
    C3 -->|число| C5
    C3 -->|нет строки| C6
    C4 --> C7
    C5 --> C7
    C6 --> C7
    C7 --> KS

    KS --> K
    K -->|"проставляет done / число"| KS
    AG2 -.подтверждение обработки.-> B4
    B4 --> OS
```

## Синхронизация во времени (цикл 30 минут)

```mermaid
sequenceDiagram
    participant W as Зал / Сайт / Телефон
    participant A1 as Агент 1
    participant OS as Таблица заказов
    participant A2 as Агент 2
    participant KS as Kitchen assistant
    participant K as Кухня

    W->>A1: Новый заказ (в любой момент)
    A1->>OS: Строка, статус = new

    loop Каждые 30 минут
        A2->>OS: Читает строки status = new
        A2->>A2: Суммирует Quantity по блюду
        A2->>KS: Обновляет B и C по правилам
        A2->>A1: Подтверждение: заказы обработаны
        A1->>OS: status new → redirected
        K->>KS: Смотрит список, готовит блюда
        K->>KS: Проставляет Total status (done / число)
    end
```

## Почему именно так

- **Разделение прав на запись.** Агент 2 только читает исходную таблицу и пишет в kitchen assistant; Агент 1 — единственный, кто меняет статус в исходной таблице. Это исключает состояние гонки (race condition), когда два процесса одновременно правят один и тот же статус.
- **`redirected` выставляется только после подтверждения.** Если Агент 2 упадёт на середине цикла, необработанные заказы останутся в статусе `new` и будут подхвачены в следующий раз — данные не теряются.
- **Столбец C ("Total status") пишет только кухня**, кроме двух моментов: создание новой строки и реактивация строки из `done` — тогда Агент 2 подставляет `new`, чтобы сигнализировать "по этому блюду снова есть, что готовить".
